# Overview of LLM based Multi-Agents

The **LLM-based multi-agent system** consists of specialized **agents** with different **roles**. Some of the agents may use single-agents or leverage external tools to fulfill their *Service Level Objectives* (**SLO**), but the overall behavior of the system is determined by communication, orchestration and the quality of the model. According to the *common surveys*$^{[1]}$, errors and performance degradations can propagate between agents leading to the *Cascading effects* and *logical errors*.

We considered several multi-agent systems like **Codex**$^{[2]}$, **OpenManus**$^{[3]}$ and **CrewAI**$^{[4]}$, that demonstrate the transition from using LLM as a single model to agent-based systems in which LLM acts as an active component, performing actions, interacting with the environment and other agents:

| Tool | Remarkable features |
| :--- | :--- |
| Codex | • Autonomous LLM agent<br>• Possibility of considering long-term and parallel tasks |
| OpenManus | • Integration of external tools and APIs<br>• Well-illustrates dependency errors |
| CrewAI | • Explicit agent orchestration<br>• Deterministic/LLM-based logic capability<br>• Convenient for scenarios and experiments |

All these multi-agent systems are **open-source** and thus **easy to research**. Yet, these systems essentially have a **focus on functionality**, not reliability (fault handling only done on lower abstraction levels). Thus, there is **no availability analysis** done for these systems so as **no ready-to-use fault-models** and sub-health benchmarks.

# Multi-agent system stack: Inference: KV-cache

The $VLLM^{[7]}$ framework is a standard inference engine widely used in production. On a small scale it handles all *control* and *scheduling* to leverage the inference performance. It supports parallelisms out of the box, but the key its feature is **Paged Attention**:

**Paged Attention** borrows concepts from *virtual memory*:
*   Tokens $\approx$ bytes
*   KV-cache $\approx$ process address space
*   KV-block $\approx$ page
*   Request $\approx$ process

Thus, requests use logical KV-blocks that are stored in non-contiguous physical GPU memory blocks (*as on the picture*):

| Paged attention pros | Paged attention cons |
| :--- | :--- |
| Significant GPU memory savings | Complex attention implementation |
| Elimination of memory fragmentation | Small runtime overhead |
| Support for efficient dynamic batching | Doesn't solve distributed inference |

> 📊 **FIGURE [8]**: KV cache paging
>
> **Визуальный анализ**:
> - **Тип**: Схема распределения памяти (Mapping).
> - **Оси**: Логические блоки (Logical KV blocks) сопоставляются с физическими блоками (Physical KV blocks).
> - **Легенда**: 
>   - **Request A** (желтый): содержит токены "Four score and seven", "years ago our fathers", "brought".
>   - **Request B** (зеленый): содержит токены "It was the best", "of times".
> - **Ключевой тренд**: Демонстрируется, как логически последовательные данные запросов (A и B) разбиваются на блоки и сохраняются в физической памяти GPU в произвольном (непрерывном) порядке, аналогично подкачке страниц в ОС. Физический блок 7 содержит продолжение данных Request A, хотя логически они идут в начале.

On the larger system's scale the Paged Attention mechanism allows to distribute the calculations and let *control & scheduling* processes to be handled by a *dedicated layer*.

### [11]

# Multi-agent system stack: Inference: PD-separation

**Prefill/Decode instances (PD instances)** – are dedicated inference instances for two stages of the request processing - ***Prefill*** (processing the prompt/prefix) and ***Decode*** (individual token generation).

**The need to separate Prefill and Decode:**
These stages have different load profiles and different bottlenecks: Prefill is often limited by computations, Decode – by latency/memory/bandwidth. Having separate prefill and decode instances allow to **control**, **plan** and eventually **balance** the system load in the system leveraging the performance and hardware utilization.

**What needs to be managed and controlled at the PD level:**
*   ***Routing requests between pools***: where to send Prefill and where to continue Decode.
*   ***Admission control***: limiting the incoming flow to maintain SLO and avoid queue bloating.
*   ***SLA-aware planning***: rules that take into account TTFT/TPOT, tail-latency, queue lengths, traffic type (short/long requests).
*   ***Independent pools scaling***: The Prefill pool and Decode pool are scaled separately according to their metrics.
*   ***Status and KV-cache***: deciding where to store KV and how to minimize “expensive” state transfers between nodes/pools.

### [12]

# Multi-agent system stack: Scheduling PD-instances

To schedule and manage PD-instances in the cluster the Ray framework is used. *Ray* — is a distributed computing orchestration framework, that creates an abstraction layer over the physical cluster, providing primitives (**Actors**, **Tasks**) for state and resource management. It is the infrastructure basis for most modern inference systems (**vLLM**, **Alpa**, **KubeRay**):

| Architectural component | Function in a distributed system | Importance for accessibility modeling |
| :--- | :--- | :--- |
| Global Control Store (GCS) | Centralized storage of metadata and sharding tables. The "brain" of the cluster. | Determines the stability of the Control Plane. Allows modeling scenarios of control loss and state desynchronization. |
| Distributed Object Store (Plasma) | Shared Memory with zero-copy access for processes on the node. | Critical for prefill/decode decoupling. Models the physics of heavy key-value cache transfers without the overhead of serialization. |
| Raylet & Heartbeats | Local scheduler and heartbeat check mechanism (keep-alive). | Defines the $T_{detected}$ metric (failure detection time). Creates a "blind spot" when requests are sent to an already unavailable node. |
| Placement Groups | Atomic resource allocation for a group of related actors (Gang Scheduling). | A basic primitive for Expert Parallelism. Explains the mechanics of cascading failure: the failure of one GPU disables the entire group of MoE experts. |

### [13]

# Multi-agent system stack: Scheduling traffic

The processing time of user's traffic is inherently unpredictable in terms of resource and latency. To address this problem the scheduling is used and the availability of the whole system depends on it dramatically. One of the few open-source schedulers, that handles this influence is **Llumnix**$^{[9]}$. It introduces token-level scheduling and preemption for the first time with the ***KV-cache*** migration:

**Preemption** - is an interruption of the current task to give resources to another, higher-priority task. This means:

*   Token generation for a single request can be temporarily suspended.
*   The $KV$-cache state is saved or migrated.
*   The GPU then begins working on a more important request.
*   When resources are free again, the interrupted task resumes from where it left off.

**Token-level scheduling**$^{[10]}$ - is an approach to scheduling tasks at the level of individual tokens, rather than at the level of entire requests. With token-level scheduling, the system can:

*   Split text generation into individual tokens.
*   Process tokens from different requests in parallel.
*   Dynamically switch between tasks.

| Token-level scheduling pros | Token-level scheduling cons |
| :--- | :--- |
| Significant reduction in tail latency | $KV$-cache migration - expensive |
| More stable SLA | High complexity of implementation |

# Multi-agent system stack: Control

**Kubernetes**$^{[11]}$ - a platform for managing containerized workloads and services through declarative configuration and automation. The core concept is to abstract infrastructure (servers, networks, storage) and allow *service-based system management*, where developers declaratively describe the desired application states.

The cluster architecture is based on the separation of roles:

*   **Control Plane (Master-nodes)** - the control plane, which includes the API server (kube-apiserver), controllers (kube-controller-manager), scheduler (kube-scheduler), and state store (etcd). It is responsible for making decisions and maintaining the desired state.
*   **Worker Nodes** - machines running applications. Each node contains a container runtime (e.g., Docker or container), a Kubelet agent for communicating with the Control Plane, and a kube-proxy proxy component for network routing.

**Kubernetes allows to provide**: Orchestration, automation, control resource utilization, scaling, isolation and fault recovery.

The above makes ***Kubernetes*** a convenient platform for deploying and managing large language model (LLM) inference using ***vLLM+Ray***.

# Dynamic Availability

Modern LLM inference systems serve diverse workloads: interactive applications, batch processing, and multi-agent systems. The latter creates particularly challenging patterns - agents retry failed requests, chain multiple API calls, and generate correlated bursts. This makes strict time guarantees critical for system stability.

**Definition:**
Dynamic Availability measures the fraction of requests that meet a specified time threshold.

$$
DA = P(T_{total} \leq T_{threshold})
$$

where:
*   $T_{threshold}$ - threshold time

**Measured from simulation:**
$$
DA = \frac{N_{success}}{N_{total}}
$$

where:
*   $N_{success}$ - requests meeting the time threshold
*   $N_{total}$ - total number of requests

By Law of Large Numbers, the measured ratio converges to the probability as $N \to \infty$.

**Practical use:**
*   Direct alignment with time-based service requirements
*   Quantitative comparison of system configurations
*   Basis for identifying reliability bottlenecks

Example: $DA = 0.95$ means 95% of requests are served within time limit.

To improve DA, we must identify which components to optimize $\rightarrow$ Sensitivity Analysis

# Sensitivity research: Classical statistics

**Background: Classical Reliability Models.** As part of our research, we reviewed classical approaches to reliability modeling to understand existing methodologies.

**Operational readiness ratio$^{[16]}$.** The probability that the system will be operational at any given moment in time and, from that moment on, will continue to be operational for a specified period of time.

| Pros | Cons |
| :--- | :--- |
| • Comprehensiveness.<br>• The assessment takes into account both the system's readiness at a random moment and its ability to perform the task thereafter. | • Time dependence.<br>• The assessment depends on the choice of the review interval, which complicates comparison of systems. |

**Probability of system failure-free operation$^{14}$.** For a top-level element, the probability of system failure is calculated as the product of the probabilities of failure of each of the preceding elements.

| Pros | Cons |
| :--- | :--- |
| • Ease of calculation.<br>• Easily calculated for both serial and parallel structures. | • Static.<br>• The assessment does not take into account changes in failure rates over time. |

### [21]

# Sensitivity research: Nelson reliability model

There are various models of system reliability. The following were considered in this study:

**Measuring models.** Based on measuring reliability properties for a fixed software configuration. It is assumed that detected errors are not corrected.

**Nelson model**$^{[17]}$. The Nelson model estimates software reliability as the probability of failure-free operation during a given test run, calculated by dividing all possible program inputs into disjoint domains, with each domain assigned a probability of being used in real-world operation. The final reliability estimate is obtained by summing, over all domains, the product of the probability of selecting input data from a given domain and the frequency of successful test runs in that domain.

| Pros | Cons |
| :--- | :--- |
| • Nelson's model takes into account not only the number of remaining faults, but also how often the fault-causing module is used. | • The model is subject to the curse of dimensionality. The number of paths grows exponentially as the number of branches increases. |

### [22]

# Sensitivity research: Musa reliability model

**Evaluation models** are based on a series of test runs when testing systems in a real environment and depend on the frequency of occurrence and elimination of causes of failures and their intensity.

**Musa model**$^{[18], [19]}$. The idea behind the model is that software reliability is directly related to the number of errors remaining in the code and can be quantified by the failure rate, measured in CPU time. The model postulates that as errors are discovered and corrected during testing, the failure rate decreases linearly, and the mean time between failures increases accordingly. It takes into account the intervals between failures, the time between failures is distributed exponentially, and the failure rate is proportional to the number of errors.

| Pros | Cons |
| :--- | :--- |
| • Simplicity and computational efficiency. Parameter estimation does not require complex calculations. | • The model does not take into account the heterogeneity of failures. Due to the assumption of linear and perfect error correction, the model may predict that high reliability levels will be reached more quickly than is actually the case. |

### [23]

# Sensitivity research: Musa-Okumoto reliability model

**Musa–Okumoto model (logarithmic)**$^{[20], [21]}$. The model describes the process of detecting failures over time, where the average number of failures is not a linear function.

The mathematical expectation of the errors found is: $m(t) = \frac{1}{\theta} \ln(\lambda_0 \theta t + 1)$, where $\lambda_0$ – initial failure rate, $\theta$ – the rate of decline of the failure rate with each eliminated defect, and the failure rate function $\lambda(t)$ is: $\lambda(t) = \frac{\lambda_0}{\lambda_0 \theta t + 1}$, $m(t)$ and $\lambda(t)$ depend on time.

| Pros | Cons |
| :--- | :--- |
| • Unlike models that assume a finite (and fixed) number of errors in the code, the Musa-Okumoto model allows that theoretically the number of failures can grow indefinitely, which in practice reflects reality. | • The model treats all failures as homogeneous events and does not take into account the modular structure of the program and the fact that different parts of the code may have different reliability. |

**Our approach.** Classical models focus on software reliability during testing phase. Our work addresses runtime availability of distributed inference systems $\rightarrow$ Dynamic Availability metric (next).

### [24]

# Sensitivity definition

Given DA metric, we need to identify which system components have the greatest impact on availability.

**Definition:**
Sensitivity measures how DA changes when a component parameter is varied.

$$
S_{i}^{(\theta)} = \frac{\partial DA}{\partial \theta_{i}}
$$

where:
*   $\theta_{i}$ - parameter of component $i$ ($T, P_{err}, D_{delay}, etc.$)
*   $S_{i}^{(\theta)}$ - sensitivity index

Practical use:
*   Prioritize optimization efforts
*   Identify reliability bottlenecks
*   Guide resource allocation decisions

**Example application:** If $S(P_{err}, Decode) = -0.4$, then reducing Decode error rate by $10\%$ improves DA by approximately $4\%$.

For stochastic simulators, analytical derivatives are unavailable. Naive numerical methods require prohibitive computational cost $\rightarrow$ specialized SA methods needed.

### [25]

# Sensitivity calculation complexity

**Classical numerical differentiation:**

$$
S_i \approx \frac{DA(\theta_i + \Delta) - DA(\theta_i - \Delta)}{2\Delta}
$$

**Why this fails for stochastic simulators:**

**Problem 1: Statistical noise**
Each $DA$ estimate has variance $\sigma^2 \approx \frac{DA(1-DA)}{N}$. For $DA \approx 0.95$ and $N = 10'000$ runs: $\sigma \approx 0.002$. Parameter effects smaller than $\sim 2.0\sigma \approx 0.004$ (0.4% change in $DA$) are indistinguishable from noise at 95% confidence.

**Problem 2: $\Delta$ selection dilemma**
*   Small $\Delta \rightarrow$ noise dominates signal
*   Large $\Delta \rightarrow$ poor estimate of derivative

**Problem 3: Computational cost**
For $P$ parameters: $2P$ evaluations needed. Each evaluation requires $N$ simulations for reliable results.

Example: $P = 10, N \approx 10'000$ (typical) $\rightarrow$ 200,000 total simulations.

**Solution:** Specialized SA methods (Morris, PAWN, Sobol)
These methods use structured sampling strategies to extract more information per simulation, reducing computational cost by orders of magnitude compared to the naive approach.

### [26]

# Sensitivity Analysis: Methods Overview

### Why SA for stochastic simulator?
**Problem:** Finite difference method is unstable for Monte Carlo simulations due to noise.
**Solution:** Use specialized SA methods designed for stochastic models.

### Three-stage pipeline for sensitivity analysis:
**Stage 1 - Morris Screening:**
Rapidly evaluates all parameters using elementary effects. Computes $\mu^*$ (mean absolute effect) and $\sigma$ (standard deviation) for each parameter. Parameters with low $\mu^*$ are filtered out, reducing computational cost for subsequent stages.

**Stage 2 - PAWN Analysis:**
For remaining parameters, computes sensitivity indices by measuring how DA distribution changes when a parameter is fixed. Generates unconditional sample ($N=3000$ simulations with all parameters varying), then conditional samples for each parameter (fixing one parameter at different values).
**Output:** ranked sensitivity indices showing which parameters most affect DA.

**Stage 3 - RSA Diagnostic:**
Splits simulation results into "success" and "failure" groups based on DA threshold. Identifies parameter ranges that lead to failures.
**Output:** operational limits (e.g., "keep $P_{err} < 0.06$").

---

### Methods comparison:

| Method | Measures | Cost | Use case |
| :--- | :--- | :--- | :--- |
| *Sobol* | Variance decomposition | High | Interactions |
| *Morris* | Elementary effects | Low | Screening |
| *PAWN* | CDF shift | Medium | Threshold metrics |
| *RSA* | Group separation | Zero* | Diagnostics |

**Conclusion:** For threshold metric $DA = F(T_{threshold})$, PAWN is more directly related than Sobol.

### [27]

# Sensitivity Analysis: Morris and PAWN methods

---

### Morris Screening
**Elementary Effect:**
$$
EE_i = \frac{f(X + \Delta \cdot e_i) - f(X)}{\Delta}
$$

**Metrics:**
$$
\mu_i^* = mean(| EE_i |), \sigma_i = std(EE_i)
$$

**Morris Interpretation:**
*   *High* $\mu^*$, *low* $\sigma \rightarrow$ *important, linear effect*
*   *High* $\mu^*$, *high* $\sigma \rightarrow$ *important, nonlinear/interactions*
*   *Low* $\mu^* \rightarrow$ *can be discarded*

**Output:** Ranked list of parameters by $\mu^*$ value.

**Cost:**
$N = r \cdot (k + 1)$, where $r = 30, k = 11 \rightarrow 360$ runs

---

### PAWN Analysis
Key idea: $DA = F(T_{threshold})$ is a CDF value. PAWN measures CDF shift.
$$
S_i = median_{x_i} \left( \max_{y} | F_Y(y) - F_{Y|X_i}(y) | \right)
$$

**PAWN algorithm:**
1.  Generate $N_u$ unconditional samples (vary all parameters)
2.  For each parameter $i$: split its range into $S$ slices
3.  For each slice: generate $N_c$ conditional samples (fix parameter $i$, vary others)
4.  Compute KS-statistic (maximum difference between unconditional and conditional distributions)
5.  $S_i = median(KS \text{ across slices})$

**Output:** Sensitivity indices $S_i$ for each parameter, ranked by magnitude.

**Cost:**
$N_u + k' \cdot S \cdot N_c = 3000 + 6 \cdot 10 \cdot 200 = 15000 \text{ runs}$

# Sensitivity Analysis: RSA method and Integration

### RSA (Regional Sensitivity Analysis) workflow:
1. Use simulation results from PAWN stage ($N=3000$ samples)
2. Set target threshold (e.g., $DA_{target} = 0.99$)
3. Split samples:
   *Behavioral* ($DA \ge target$) vs *Non-behavioral* ($DA < target$)
4. For each parameter: compute KS value (separation metric between success/failure groups, 0-1 scale)
5. Identify parameter ranges where failures concentrate

0 additional runs (reuses PAWN sample)

### RSA interpretation:
* **High KS** $\rightarrow$ parameter strongly separates success from failure
* Compare distributions to find critical ranges
* Example output: "When $P_{err\_dc} > 0.08$, 73% of requests fail"

### Integration benefit:
PAWN identifies WHICH parameters matter. RSA identifies WHAT VALUES cause problems. Together: complete actionable guidance.

*   $s \in V$ - input node (query source)
*   $t \in V$ - output vertex (calculation result)

Defines node behavior when receiving a request that failed in predecessor. Map[ErrorType $\rightarrow$ Reaction]:
*   **IGNORE** - error does not affect processing, continue normally without penalty
*   **DELAY** - add penalty time $D_{delay}$ before processing, then continue
*   **RECOVERY** - enter recovery mode for $D_{recovery}$ time, then resume processing

*   **TIMEOUT** - processing time exceeded limit $T$
*   **CACHE_MISS** - requested data not in cache
*   **DECODE_ERROR** - transformation failure
*   **PARTIAL_RESULT** - incomplete output

### Error routing (`out_ML`)
Map[ErrorType $\rightarrow$ BlockID] - routing table.
When error occurs, determines target node for recovery via FALLBACK edge.

`out_ML` = {CACHE_MISS $\rightarrow$ `V1_LB`}
When cache miss occurs, request routed to load balancer

`V7_Agg`: `in_ML` = {PARTIAL_RESULT $\rightarrow$ DELAY}
Waits additional time when receiving partial results from experts

>   1. **SUM (Sequential)**: Последовательная обработка (A → B → C). Общее время $T = T_1 + T_2 + T_3$.
>   2. **MIN (First wins)**: Параллельная обработка с выбором первого результата. Общее время $T = \min(T_B, T_C)$.
>   3. **MAX (Wait for all)**: Барьерная синхронизация. Общее время $T = \max(T_1, T_2, T_3)$.
>   4. **QUEUE (With Queuing Effects)**: Модель с учетом очереди. Общее время $T_{total} = T_{waiting} + T_{processing}$.
> - **Ключевой тренд**: Иллюстрация того, как топология соединений и логика узла определяют итоговую задержку системы.

## Aggregation Type Definition

When a node receives data from multiple predecessors (fan-in topology), the system must determine how to compute the total processing time.

Aggregation type is a node parameter that defines the rule for combining incoming processing times into a single value.

Different system architectures require different aggregation semantics:
*   Sequential pipelines use **SUM**
*   Redundant systems use **MIN**
*   Barrier synchronization uses **MAX**
*   Resource-constrained nodes use **QUEUE**

These graph concepts model distributed systems. Evaluating their availability requires a simulation platform.

***

### Detailed Logic of Aggregation Types:

#### 1. SUM (Sequential)
*   **Graph**: $A \xrightarrow{T_1} B \xrightarrow{T_2} C \xrightarrow{T_3}$
*   **Formula**: $T = T_1 + T_2 + T_3$
*   **Use case**: Pipeline processing (tokenize $\rightarrow$ encode $\rightarrow$ decode).

#### 2. MIN (First wins)
*   **Graph**: Parallel paths to node $D$ from $B$ and $C$.
*   **Formula**: $T = \min(T_B, T_C)$
*   **Use case**: Hedged requests (PARALLEL edges + MIN aggregation).

#### 3. MAX (Wait for all)
*   **Graph**: Router to multiple experts (Exp1, Exp2, Exp3) aggregating in 'Agg'.
*   **Formula**: $T = \max(T_1, T_2, T_3)$
*   **Use case**: MoE architecture (wait for all activated experts).

#### 4. QUEUE (With Queuing Effects)
*   **Logic**: Accounts for contention between requests.
*   **Formula**: $T_{total} = T_{waiting} + T_{processing}$
*   **Critical Condition**: $\Delta$ When $\lambda \to \mu$ (high load): $T_{waiting} \to \infty$
*   **Use case**: Load balancers, GPU schedulers, token bucket rate limiters.

# Conclusions on stage 1.2

An architecture for modeling dynamic availability of distributed inference systems has been built. At its core - four elements. The first of them, the configuration generator, transforms system specifications into formal representation. The intermediate representation stores the graph structure, while the simulation engine processes the request flow. The analysis module detects bottlenecks.

At the same time, aggregation methods are implemented for time calculation: sequential, parallel, with queueing.

Three methods form the sensitivity analysis pipeline. The Morris Screening method filters out minor parameters. Further, the PAWN Analysis method determines sensitivity indices, while the RSA Diagnostic method identifies critical parameter ranges.

The platform solves three tasks: finds critical components, lets you choose parameters for optimization, estimates expected availability improvement. Thus, modeling results become the foundation for making decisions about architecture optimization.
